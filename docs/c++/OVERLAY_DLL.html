OverlayDll：挂钩与每帧合成

本文说明 OverlayDll.dll 如何进入宿主进程、钩住 DXGI Present / Present1，以及在每次 Present 之前把共享内存里的 Qt 像素画到当前后缓冲上。实现文件：OverlayDll/dllmain.cpp、d3d11_hook.cpp、overlay_renderer.cpp、shared_surface.cpp。

1. 导出接口（宿主调用）

导出 作用
Overlay_Start() 安装 Present / Present1 钩子（内部 InstallPresentHook）。
Overlay_Stop() 移除钩子并 InvalidateDevice（释放 D3D 资源与共享内存读取器）。
Overlay_SetDrawEnabled(int) 非 0：正常读共享内存并合成；0：挂钩仍执行，但跳过 Qt 纹理路径（便于只测钩子或省电）。
Overlay_SetDebugHookTint(int) 非 0：不读共享内存，画半透明纯色全屏三角（验证 MinHook + 状态保存/恢复是否工作）。

入口在 dllmain.cpp：Overlay_Start 仅调用 InstallPresentHook()。

2. 为什么钩 Present

游戏的 交换链每帧结束时调用 IDXGISwapChain::Present（或 IDXGISwapChain1::Present1）。在 原始 Present 被调用之前插入逻辑，可以把 overlay 画进 当前即将提交的后缓冲，玩家看到的就是「游戏画面 + UI」的合成结果。

要点：

  • 钩子是 进程内的：DLL 被 LoadLibrary 进 TestGame 后，与游戏共用同一 D3D11 设备和上下文（通过 swapChain->GetDevice / GetImmediateContext 取得）。
  • 必须在 Present 路径里做合成，而不是随意线程里画：否则目标 RT 与帧边界不对齐，容易出现闪屏或画在错误缓冲上。

3. 如何拿到 Present / Present1 地址

游戏自己的交换链在运行时才会创建，但 vtable 里 Present 槽位对标准 DXGI swap chain 是稳定的。工程里用 临时设备算一次函数指针（GetPresentAddresses）：

  1. 注册临时窗口类，创建 2×2 隐藏窗口。
  2. D3D11CreateDeviceAndSwapChain 得到 IDXGISwapChain*。
  3. vtable[8] → Present（与 DXGI 接口布局一致）。
  4. QueryInterface(IDXGISwapChain1)，取 Present1（代码里为 vtable 下标 22，注释中推演了 IUnknown → … → Present1 的偏移）。
  5. 销毁临时窗口与设备。

这样得到的是 该 DXGI 版本下 Present 在虚表中的地址；MinHook 对该地址下钩后，任意进程内随后创建的 swap chain，只要走同一实现，都会进入 HookedPresent / HookedPresent1。

4. MinHook 安装顺序

InstallPresentHook（d3d11_hook.cpp）大致步骤：

  1. MH_Initialize
  2. MH_CreateHook(present, HookedPresent, &g_originalPresent) + MH_EnableHook
  3. 若 Present1 地址存在且与 Present 不同，再对 Present1 单独 CreateHook + EnableHook

被钩函数里先做 overlay::RenderOverlay(sc)，再调用 保存下来的原始函数（g_originalPresent / g_originalPresent1），保证游戏原有 Present 行为不变。

HookedPresent1 里把 IDXGISwapChain1* QueryInterface 成 IDXGISwapChain* 再交给 RenderOverlay，这样 纹理上传与画四边形只实现一份。

RemovePresentHook 中 DisableHook / RemoveHook 后 MH_Uninitialize，并调用 overlay::InvalidateDevice()。

5. RenderOverlay 一帧在做什么

逻辑在 overlay_renderer.cpp 的 RenderOverlay(IDXGISwapChain* swapChain)。

5.1 设备与资源

  1. swapChain->GetDevice(IID_ID3D11Device)，GetImmediateContext。
  2. EnsureResources(device, ctx)：若当前 g_res 绑定的 device 变了（或首次），会 重建 VS/PS、采样器、混合、光栅（含 scissor）、深度模板（深度关闭）、动态常量缓冲、以及 SharedSurfaceReader::Open()。
  3. 调试路径：若 g_debugHookTint → DrawSolidDebugOverlay（全屏三角 + 青色 PS），然后 return。
  4. 关闭绘制：若 g_drawEnabled == 0，直接 return。

5.2 读共享内存

  1. g_res->reader.LockForRead(&hdr, &pixels)（见 shared_surface.cpp）：OpenFileMapping / MapViewOfFile(…, 0)、WaitForSingleObject(mutex)，校验宽高、stride、映射长度。
  2. 若 hdr->overlayVisible == 0，解锁并 return（与 OverlayUI 里复选框一致）。

5.3 CPU 像素 → GPU 纹理

  1. EnsureTexture(w, h)：按需创建 DYNAMIC R8G8B8A8_UNORM 纹理 + SRV。
  2. Map 纹理，memcpy 每行：源 stride 用 hdr->stride（与 OverlayUI 紧密打包一致），目标用 mapped.RowPitch（可能大于 w*4）。
  3. Unmap，UnlockRead() 释放互斥量。

5.4 画到后缓冲（居中 1:1 四边形）

  1. GetCurrentBackBufferIndex(swapChain)：
    - 单缓冲或非 flip 链 → 0；
    - flip 模型 → 尽量用 IDXGISwapChain3::GetCurrentBackBufferIndex，与游戏实际将要 Present 的缓冲一致（否则可能画到「另一张」缓冲上看不见）。
  2. GetBuffer(bbIndex) → 临时 RTV，绑定为唯一 render target（深度设为 nullptr）。
  3. 保存/恢复一大块管线状态：RTV/DSV、viewport、scissor、blend、RS、DSS、VS/PS/GS、SRV、 sampler、input layout、topology、VB/IB、VS 常量缓冲槽 0 等。
    - 若某状态原为 null（例如 RS），恢复时 RSSetState(nullptr)，避免把无效状态留回游戏。
  4. 设置全屏 viewport + 全屏 scissor、预乘 alpha 混合、关闭深度测试。
  5. 常量缓冲写入 OverlayQuadCB：origin / quadW/H 为 后缓冲上居中 的像素矩形（逻辑尺寸不超过 w,h 且不超过后缓冲宽高），backbufferW/H 供 VS 转 NDC。
  6. VS vsOverlayQuad：生成 6 个顶点的矩形，按 像素对齐 放到屏幕；PS 用 sampPoint 点采样纹理，避免小图拉伸糊成一片。
  7. Draw(6, 0) 后按保存值 全部还原，释放临时 RTV。

6. 与 OverlayUI 的约定

  • 共享内存名 / 互斥量名：common/overlay_ipc.h 与 OverlayUI 一致。
  • 像素布局：SharedOverlayHeader 后紧跟 RGBA，行紧密 width*4（DLL 读 hdr->stride）。
  • 可见性：overlayVisible 为 0 时不画（仅透明/跳过）。

语义管道（GameOverlaySemantics）由 OverlayUI 单独提供；OverlayDll 当前不消费语义，只消费位图。

7. 调试与日志

  • Overlay_SetDebugHookTint(1)：确认钩子与 D3D 状态恢复是否正常（不依赖 Qt）。
  • 日志文件：部分失败路径会追加 %TEMP%\GameOverlayDemo_dll.log（如 LockForRead、纹理创建失败等）；shared_surface.cpp 里 OpenFileMapping / MapViewOfFile 等也会 单次记入，避免刷屏。

8. 风险与限制（心里有数）

  • 多设备 / 多 swap chain：当前实现按「单 immediate context + 每次 Present 的链」工作；若游戏多窗口或多设备，需要扩展「按 swapChain 存 g_res」等策略。
  • Present1 与 Present 并存：对部分运行库两者都会走，已尽量分别挂钩；若仍遇特例需结合具体游戏验证。
  • 反作弊 / 完整性：进程内挂钩属于敏感能力，商业游戏环境需单独评估。

9. TestGame：谁在何时调用 Overlay_Start

本 demo 里，只有 TestGame.exe 会加载 OverlayDll.dll；调用链在 TestGame/main.cpp 的 LoadOverlayDll()。

9.1 启动顺序（wWinMain）

  1. 创建窗口、InitD3D：创建 D3D11 设备与交换链（之后每帧 PresentFrame 里会调用 g_swap->Present）。
  2. LoadOverlayDll()：在 D3D 已就绪 之后立刻执行（仍在进入消息循环之前）。
  3. EnsureOverlayUiRunning()：如需要则拉起 OverlayUI.exe（共享内存生产者）。
  4. StartOverlaySemanticsReaderThread()：连接语义管道（与 DLL 无关）。
  5. 进入主循环：每帧 PresentFrame → Present → 已被钩住则先走 RenderOverlay，再进游戏原本的 Present。

这样保证：第一次 Present 发生时，MinHook 已经安装（否则首几帧可能漏画 overlay）。

9.2 LoadOverlayDll() 做了什么

  1. 用 GetModuleFileNameW 得到当前 exe 路径，把文件名换成同目录下的 OverlayDll.dll（避免依赖当前工作目录；与 CMake post-build 拷贝位置一致）。
  2. LoadLibraryW、GetProcAddress 解析：
    - Overlay_Start
    - Overlay_SetDrawEnabled
    - Overlay_SetDebugHookTint
  3. 若 Overlay_Start 非空，立即调用 → 内部 InstallPresentHook()。
  4. 将 g_drawOverlay 设为 true，并调用 Overlay_SetDrawEnabled(1)，使默认进入游戏即可看到 Qt 合成（无需再按一次热键）。

9.3 运行期热键（仍通过同一函数指针）

按键 行为
F1 仅把 OverlayUI 窗口前置（BringOverlayUiWindowToForeground），不关合成。
F4 切换 g_drawOverlay，并调用 Overlay_SetDrawEnabled(0/1)（挂钩仍在，DLL 内跳过纹理路径）。
F3 切换 Overlay_SetDebugHookTint（纯色调试路径，验证钩子与 D3D 状态）。

鼠标在面板区域内时，仍由 WndProc 通过管道发输入给 Qt（与 Overlay_Start 无直接调用关系）。

9.4 退出时

主循环结束后依次 ReleaseD3D、ShutdownOverlayUiIfLaunchedByUs，然后 FreeLibrary(g_overlayDll)。

当前 demo 未在卸库前显式调用 Overlay_Stop()；若需严格卸钩再释放，可在 FreeLibrary 前 GetProcAddress("Overlay_Stop") 并调用（与你们工程策略一致即可）。
